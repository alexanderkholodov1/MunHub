# MunHub Lab v6.0 — In-house AI design (planning)

> Depends on: `00`–`02` + the physics theoretical report. Covers EPIC-10 (S39–S41).
> **Design only for now.** Implementation in F7, once Red Clara provides a server.
> Decision D12: **classical ML first, designed to scale to deep learning.**
> The physicist agent has veto power over any scientific claim generated by the AI.

---

## 1. Goal and philosophy

An AI that is **simple but functional and useful**, trained on our own data, that: surfaces
statistics and insights, attempts to explain events, forecasts, retrains autonomously
(self-improve), self-repairs (self-heal), and helps users understand what the values mean and
which are "normal". **No paid APIs**: an in-house model on the server.

Guiding principle: **honest utility**. With 1-SiPM detectors and modest data volumes,
well-built classical models deliver more value (and more interpretability) than premature DL.
The pipeline is designed so DL can be added later without rebuilding it.

---

## 2. Available data (hardware reality)

From `minute_records`: `ec, cc, sm, sx, sn, tp, pr, dt` + derived (`ec_dt`, `ec_corr`, `flux`).
From `external_events`: NMDB, NOAA SWPC, DONKI, Dst/Kp. Detector metadata (lat/lon/altitude,
placement). **Limitation:** 1 SiPM cannot separate particle types → the AI does NOT classify
particles in phase 1; it works on rate, amplitude spectrum, and correlations.

> **Hard constraint (Poisson, `THEORETICAL-FOUNDATION.md` §10):** the minute is too noisy
> (error ~1.6%). All anomaly/Forbush detection operates on **windows of hours**
> (N≥100k counts, error <0.3%) and **persistent ≥3σ** thresholds, on the already-corrected
> rate (dead time → barometric → thermal). No "anomalies" on a single minute.

---

## 3. Capabilities (models), from simple to advanced

| # | Capability | Approach (phase 1, classical) | Value |
|---|-----------|---------------------------|-------|
| C1 | **Baseline / "normal values"** | Robust per-detector statistics (medians, IQR, diurnal/seasonal profiles); Poissonian nature (√N) | Defines what is normal per site |
| C2 | **Barometric/thermal correction** | Non-linear **Gradient Boosting (XGBoost/LightGBM)** with **local** β, outperforming the static linear fit; dead-time correction first | Removes the atmospheric effect (key) |
| C3 | **Anomaly detection** | Autoencoder / Isolation Forest over the vector (mV, corrected rate, P, T); robust z-score / STL | Alerts on rare events |
| C4 | **Forbush detection** | Fast-drop + slow-recovery pattern in the corrected rate; corroborate with NMDB | The flagship scientific event |
| C5 | **Rate forecasting** | ARIMA/SARIMA, Prophet, or a small model; confidence band | Prediction + deviation detection |
| C6 | **Space-weather correlation** | Correlation/lag with Kp/Dst/solar wind/NMDB | Explains terrestrial↔cosmic events |
| C7 | **Insights/explanations** | Data-driven templates ("rate 12% below normal, coincides with the NMDB Forbush of …") | Clear language for users |
| (future) | Classification, DL | Temporal LSTM/Transformers, autoencoders | If GPU + data are available |

## 4. Pipeline architecture (`services/ai`, Python)

```
   minute_records ─┐
   external_events ─┼─▶ [Feature builder] ─▶ [Feature store / tables] 
   metadata ───────┘                              │
                                                  ▼
                            ┌───────── [Per-detector training] (scheduled batch)
                            │                     │ (versioned models + metrics)
                            ▼                     ▼
                     [Model registry]   ◀── [Evaluation / validation]
                            │
                            ▼
   [Inference service] ─▶ ai_insights (jsonb) ─▶ API ─▶ UI (insights/forecast/alerts)
```

- **Batch first** (scheduled jobs); optional online inference later.
- **Per detector** (each site has its own normality), with the option of global models.
- **Versioning:** every model stores `model_version`, training data/range, metrics.
- **Output contract `ai_insights`** (S41): `{detector_id, kind, ts_range, result(jsonb),
  model_version, confidence}`.

---

## 5. Self-improve and self-heal

- **Self-improve (autonomous retraining):** retraining schedule; if the new model beats the
  current one in validation (pre-defined metrics), it is promoted; otherwise it is discarded.
  Champion/challenger.
- **Self-heal:** monitoring of: data drift, metric degradation, job failures. Automatic
  actions: retry, rollback to the last good model, alert the admin, feature rebuild.
  Everything logged/auditable.
- **Guardrails:** a model never emits scientific claims outside templates approved by the
  physicist; uncertainty is always stated.

---

## 6. Technology stack

- **Phase 1 (CPU):** Python, numpy/pandas, scikit-learn, statsmodels, Prophet; orchestration
  via jobs (cron/worker); model storage on filesystem/object storage.
- **Designed to scale:** interfaces that allow substituting/adding PyTorch (DL) and GPU
  without rebuilding the pipeline (same feature store, same registry, same `ai_insights`
  contract).
- **Server:** runs on Red Clara (see tiers). No GPU in phase 1; GPU optional for DL.

## 8. Out of scope (phase 1)

- Paid LLMs/generative AI.
- Any scientific claim not backed by the theoretical report.
